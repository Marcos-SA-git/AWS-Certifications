## 1. Resumen rápido

- Una **región (*Region*)** es un área geográfica que contiene varias **zonas de disponibilidad (*Availability Zones*)**.
- Una zona de disponibilidad equivale a uno o más centros de datos redundantes.
- Una **ubicación periférica (*Edge Location*)** acelera la entrega de contenido mediante caché.
- **Modelo de responsabilidad compartida (*Shared Responsibility Model*)**: AWS asegura la **infraestructura**, el cliente asegura el **uso dentro de la nube**.
- Servicios clave a dominar: **Cómputo, Almacenamiento, Bases de Datos, Redes**.
- **Well-Architected Framework**: 6 pilares para diseñar soluciones en la nube de manera eficiente, segura y sostenible.
- **Alta disponibilidad (*High Availability*)**: diseñar sistemas que reduzcan al mínimo el tiempo de inactividad gracias a la redundancia y a mecanismos de recuperación rápida.
- **Tolerancia a fallos (*Fault Tolerance*)**: diseñar sistemas que continúan funcionando de manera normal incluso cuando fallan uno o más componentes.
- **Tiempo de Recuperación Objetivo (*Recovery Time Objective, RTO*)**: cuánto tiempo puede tardar un sistema en volver a estar operativo tras una interrupción.
- **Punto de Recuperación Objetivo (*Recovery Point Objective, RPO*)**: cuántos datos se pueden perder como máximo en caso de fallo, medido en tiempo.
- **AWS CLI (*AWS Command Line Interface*)**: herramienta de línea de comandos para interactuar con servicios de AWS.

## 2. Explicación detallada

### Regiones, zonas de disponibilidad y ubicaciones periféricas

- Una **región (*Region*)** es una localización física con **3 o más** zonas de disponibilidad, situadas en un radio de unos 100 km.
- Una **zona de disponibilidad (*Availability Zone or AZ*)** equivale a **uno o más** *centros de datos (Data Centers)* redundantes en conectividad, energía y redes.
- Una **ubicación periférica (*Edge Location*)** es un punto de red donde se almacena contenido en **caché** para entregar datos con baja latencia a usuarios finales.

Orden jerárquico: **Más ubicaciones periféricas** > **Más zonas de disponibilidad** > **Menos regiones**.

Actualmente (2025-09-20): +700 Edge Locations, 117 AZ y 37 regions.

Lista oficial de regiones y zonas: [AWS Global Infrastructure](https://aws.amazon.com/es/about-aws/global-infrastructure/regions_az/).

### Modelo de responsabilidad compartida

AWS lo llama *[Shared Responsibility Model](https://aws.amazon.com/es/compliance/shared-responsibility-model/)*. Define **qué protege AWS** y **qué protege el cliente**:

- **AWS**: seguridad **DE la nube** -> centros de datos, hardware, infraestructura y software base.
- **Cliente**: seguridad **EN la nube** -> configuración de datos, gestión de usuarios, aplicaciones y servicios.

Podemos ver a continuación un esquema de este modelo:

![Imagen 1: Shared Responsibility Model](/SAA-C03/assets/shared-responsibility-model.png)

Regla práctica:

- Si **puedes gestionarlo tú**, es tu responsabilidad.
- Si **no tienes acceso**, es responsabilidad de AWS.

> 🔹**Ejemplo**:
>
> - Tú eliges la versión de un programa o la región de despliegue.
> - No eliges el cableado de los centros de datos ni quién accede físicamente a ellos.

**Compartido**: depende del servicio. Por ejemplo -> Cifrado de datos: el cliente decide si cifrar y con qué claves, AWS garantiza que la infraestructura soporte ese cifrado.

### Servicios clave a dominar para el examen

A lo largo del curso veremos y dominaremos:

- **Cómputo (*Compute*)**: EC2, Lambda, Elastic Beanstalk.
- **Almacenamiento (*Storage*)**: S3, EBS, EFS, FSx, Storage Gateway.
- **Bases de datos (*Databases*)**: RDS, DynamoDB, Redshift.
- **Redes (*Networking*)**: VPC, Direct Connect, Route 53, API Gateway, Global Accelerator.

### Well-Architected Framework

Como su nombre indica, el "marco de trabajo para una buena arquitectura" es una guía de buenas prácticas basada en seis pilares que ayuda a diseñar y optimizar arquitecturas en AWS de forma segura, eficiente y alineada con el negocio.
Existe un conjunto de [whitepapers](https://aws.amazon.com/whitepapers/) y guías oficiales que definen las mejores prácticas en AWS. Paper recomendado: [Well-Architected Framework](https://docs.aws.amazon.com/wellarchitected/latest/framework/welcome.html?did=wp_card&trk=wp_card)

Los **6 pilares** son:

1. **Excelencia operacional (*Operational Excellence*)** -> La arquitectura funciona como debería?
    Es la habilidad para:
    - Soportar el desarrollo de las aplicaciones que usemos.
    - Ejecutar las cargas de trabajo usando las mejores herramientas apropiadas para ello.
    - Controlar con métricas que el trabajo se realiza correctamente.
    - Mejorar continuamente los procesos para ofrecer el máximo valor.
2. **Eficiencia (*Performance Efficiency*)** -> Puedo optimizar los trabajos?
    Es la habilidad para:
    - Reducir cuellos de botella o gastos innecesarios.
    - Usar los recursos adecuados para tareas concretas.
    - Mantener una eficiencia escalable ante el aumento de la demanda.
3. **Seguridad (*Security*)** -> El sistema funciona únicamente como debería?
    Es la habilidad para:
    - Proteger los datos, sistemas y activos.
    - Mantener trazabilidad de todas las acciones llevadas a cabo.
    - Aplicar seguridad por capas.
    - Proteger los datos durante el intercambio de información y en su posterior almacenamiento.
4. **Optimización de costes (*Cost Optimization*)** -> Estoy gastando mi dinero correctamente?
    Es la habilidad para:
    - Mantener las operaciones lo más baratas posible sin dejar de ofrecer el mismo valor de negocio.
    - Dejar que AWS se encargue del trabajo pesado de la infraestructura.
5. **Fiabilidad (*Reliability*)** -> Los sistemas funcionarán de manera consistente y recuperarán rápidamente en caso de desastre?
    Es la habilidad para:
    - Rendir correctamente y de manera consistente sin fallos.
    - Testear fácilmente (Test de carga/estrés, Recuperación de Backup, Recuperación ante catástrofes, etc.)
6. **Sostenibilidad (*Sustainability*)** -> Qué impacto tiene mi uso de la nube en el mundo?
    Es la habilidad para:
    - Reducir el consumo de energía.
    - Escalar correctamente los sistemas según la carga de trabajo.
    - Maximizar los beneficios de los sistemas usados.
    - Minimizar el total de los recursos usados.

Se recomienda **leer el paper al final del curso** para consolidar todo antes del examen.

Amazon pone a disposición del usuario un servicio llamado "*The AWS Well-Architected Tool*" que permite:

- Comprobar de manera consistente la arquitectura.
- Qué tan alineado estás con las mejores prácticas según AWS.
- Asistirte con la documentación de las decisiones que se toman sobre por qué y cómo se usa o no un determinado servicio o aplicación.
- Dar recomendaciones para mejorar las cargas de trabajo.
- Ayuda a diseñar y seguir soluciones que se alineen con el *Well-Architected Framework*.

### Alta disponibilidad y tolerancia a fallos

Es esencial entender la diferencia entre estos dos conceptos.

La **Alta disponibilidad (*High Availability*)**, que se caracteriza por:

- Sistemas diseñados para permanecer operativos por largos periodos de tiempo.
- Los fallos normalmente resultan en tiempos mínimos de inactividad (*downtime*) o pequeñas interrupciones (*brief interruptions*) en el servicio.
- La redundancia (*redundancy*) y los mecanismos de conmutación (*failover mechanisms*) son usados para recuperarse en caso de fallo.
- El objetivo es alcanzar una disponibilidad (*uptime*) del 99.99%.

> 🔹**Ejemplo**: Un e-commerce que empieza a tener mucho tráfico usa un balanceador de carga (*load balancer*) y auto-escalado (*auto scaling*) a través de varias zonas de disponibilidad.

La **Tolerancia a Fallos (*Fault Tolerance*)**, que se caracteriza por:

- Sistemas que siguen funcionando con normalidad incluso si uno o más componentes fallan.
- Diseñado para manejar errores de manera automática y suave.
- Los usuarios no perciben ningún tipo de interrupción.
- El objetivo es una disponibilidad del 100%.

> 🔹**Ejemplo**: El mismo e-commerce con base de datos replicada en varias regiones mediante Amazon Aurora Global, de modo que continúa operativo incluso si una región entera falla.

### Punto de Recuperación Objetivo y Tiempo de Recuperación Objetivo

Estos dos conceptos parecen similares pero son muy distintos y apuntan a objetivos diferentes.

El **Tiempo de Recuperación Objetivo (*Recovery Time Objective o RTO*)** se refiere a la máxima cantidad de tiempo que debería tardar el sistema para que se recupere en caso de que falle algún servicio o se interrumpa por algún otro motivo.
> -> ¿Qué tan rápido se necesita que estén de nuevo operativos los sistemas?

El **Punto de Recuperación Objetivo (*Recovery Point Objective o RPO*)** se refiere a la máxima cantidad de información que se puede perder por unidad de tiempo en caso de que el sistema caiga.
> -> ¿Cuántos datos se pueden perder si el sistema se cae?

### AWS CLI

El **AWS CLI (*Command Line Interface*)** permite interactuar con los servicios de AWS desde la línea de comandos, automatizando tareas y configuraciones sin necesidad de usar la consola web o la interfaz gráfica de usuario.

#### Instalación

AWS proporciona instrucciones oficiales para instalar la CLI en distintos sistemas operativos: **Linux, macOS y Windows**.  

- Se recomienda instalar **AWS CLI versión 2**, que es la utilizada en la mayoría de los cursos y demostraciones.
- Aunque se puede instalar en cualquier sistema, aprender a usar Linux para estas instalaciones es una buena práctica, ya que la nube está mayoritariamente basada en Linux y esto te prepara para entornos reales y el examen.

> 📍**Sugerencia**: Si estás en Windows puedes usar la característica de WSL para tener acceso a una terminal Linux de manera integrada sin necesidad de usar una máquina virtual que puede ser más incómoda. Para pasos detallados ve al documento de [tips](./tips.md)

**Pasos generales:**

1. Descargar la última versión desde la documentación oficial: [AWS CLI Installation](https://docs.aws.amazon.com/es_es/cli/latest/userguide/getting-started-install.html)
2. Ejecutar los pasos de instalación indicados para tu sistema operativo.
3. Verificar que la instalación se completó correctamente verificando la versión con el siguiente comando:

```bash
  aws --version
  # Respuesta aproximada esperada: aws-cli/2.30.6 Python/3.13.7 Linux/6.6.87.2-microsoft-standard-WSL2 exe/x86_64.ubuntu.24
```

> 🗒️ **Nota**: En algunos casos podrías encontrar errores relacionados con la verificación de certificados SSL. Para demostraciones, se puede omitir esta verificación, aunque **no se recomienda para entornos de producción**.

#### Configuración básica

Una vez instalada, la CLI debe configurarse para poder interactuar con tus recursos de AWS, indicando las credenciales `Access Key ID` y `Secret Access Key`, región por defecto (ej. us-east-1) y formato de salida (ej. json). Para ello ejecuta:

```bash
  aws configure
```

Tras la configuración, podrás empezar a ejecutar comandos y automatizar tareas de manera rápida y consistente.

   - Describe el papel que jugarían las Edge Locations en la entrega de contenido.

   - Resume cómo todos los conceptos anteriores (responsabilidad compartida, regiones/AZ/Edge, Well-Architected Framework) se combinan para diseñar una arquitectura básica pero sólida y alineada con las mejores prácticas de AWS.

## 4. Casos de uso

- **Regiones y zonas de disponibilidad**: elegir la más cercana a tus clientes para baja latencia.
- **Edge Locations**: entregar contenido web estático de manera ultrarrápida a nivel global.
- **Responsabilidad compartida**:
  - AWS asegura que los servidores estén siempre activos.
  - El cliente configura los permisos correctos en IAM.
- **Servicios clave**:
  - EC2 para servidores web,
  - S3 para almacenamiento de objetos,
  - RDS para bases de datos relacionales,
  - VPC para aislar y controlar redes.
- **Well-Architected Framework**: diseñar una solución con costes optimizados, resiliente y segura.
- **Alta disponibilidad (*High Availability*)**: una aplicación de comercio electrónico que distribuye el tráfico entre varias instancias en diferentes zonas de disponibilidad mediante un balanceador de carga, reduciendo el impacto si una zona falla.
- **Tolerancia a fallos (*Fault Tolerance*)**: un sistema bancario que utiliza bases de datos replicadas en múltiples regiones, de modo que incluso si una región completa se cae, las operaciones continúan sin interrupción.
- **Elasticidad y escalabilidad**: una aplicación de streaming que incrementa automáticamente la capacidad de sus servidores en horas pico y la reduce en horas de baja demanda, optimizando costos y experiencia de usuario.
- **Tiempo de Recuperación Objetivo (*Recovery Time Objective, RTO*)**: un servicio de reservas que establece un RTO de 15 minutos, garantizando que tras una caída el sistema esté nuevamente disponible en ese lapso.
- **Punto de Recuperación Objetivo (*Recovery Point Objective, RPO*)**: una base de datos financiera que define un RPO de 5 minutos, permitiendo como máximo la pérdida de transacciones de ese intervalo en caso de fallo.
- **AWS CLI (*Command Line Interface*)**: Integrar en scripts para informes simples o auditorías de infraestructura.
